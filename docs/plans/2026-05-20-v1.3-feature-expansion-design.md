# v1.3 Feature Expansion — Design

- **Date:** 2026-05-20
- **Status:** Approved, pre-implementation
- **Driver:** Differential analysis against [armeldemarsac92/csharp-lsp-mcp](https://github.com/armeldemarsac92/csharp-lsp-mcp) surfaced 9 ideas absent from our current 12-tool surface that would materially raise agent leverage on duetGPT- and XAF-shaped codebases. This design folds them into a single v1.3 release.

## Goal

Add seven tools and two cross-cutting parameter conventions to the MCP surface so that an agent can:

1. **Orient itself in a new solution** without grep heuristics (`project_overview`, `find_entrypoints`, `find_registrations`).
2. **Understand one symbol in a single call** instead of stitching 4–5 round-trips (`analyze_symbol`).
3. **Trace outbound calls**, not just inbound (`find_callees`).
4. **Map production code to its tests** (`test_map`).
5. **Spot cleanup candidates** (`find_dead_code_candidates`).
6. **Control output verbosity and noise** via a uniform `format` parameter and diagnostics filter knobs.

Performance budget: warm-up no more than 400 ms slower than v1.2 on duetGPT; new indexed queries sub-100 ms; dead-code queries 2–5 s (acceptable — run-on-demand tool, not interactive-loop).

## Non-goals

- Not adopting a persistent on-disk graph. SymbolIndex + InvocationIndex are in-process, rebuilt per Load/Reload. csharp-lsp-mcp's persisted graph amortizes their external `csharp-ls` cost; we don't have that cost.
- Not adopting their `change_impact` / `plan_change` / `verify_change` planning tools. Claude Code already orchestrates planning in-loop; a server-side reimplementation duplicates it.
- Not adopting XAML tools, completions, or code actions. Out of scope for our use cases.
- Not breaking any existing tool input/output shape. `format` is additive with a backwards-compatible default.

## Cross-cutting conventions

### `format` parameter

Added to **all** tools (existing + new). Type: `"structured" | "summary"`. Default: `"structured"` (backwards-compatible).

Implementation in `ToolBase`:

```csharp
protected enum OutputFormat { Structured, Summary }

protected virtual string BuildSummary(TResult result) =>
    throw new NotSupportedException($"{GetType().Name} has not declared a summary");

protected sealed override async Task<string> SerializeAsync(
    TResult result, OutputFormat format, CancellationToken ct)
{
    return format switch
    {
        OutputFormat.Summary    => BuildSummary(result),
        OutputFormat.Structured => JsonSerializer.Serialize(result, _opts),
        _ => throw new ArgumentOutOfRangeException(nameof(format))
    };
}
```

Each tool overrides `BuildSummary` to return one short line, e.g.:

| Tool | Summary example |
|---|---|
| `find_references` | `7 references in 4 files` |
| `hover` | `class Foo.Bar.Baz — base of FooDerived` |
| `find_entrypoints` | `3 routes, 5 middleware, 2 hosted services` |
| `find_registrations` | `12 DI registrations (3 singleton, 8 transient, 1 scoped)` |
| `analyze_symbol` | `IFooService: 7 refs, 2 impls, 0 derived, 4 callers` |
| `find_dead_code_candidates` | `5 candidates (3 high-confidence, 2 medium)` |

Errors always serialize as structured JSON regardless of `format` — error envelopes are machine-parseable.

### Filter knobs

**Diagnostics tools** (`get_compilation_errors`, `get_document_diagnostics`) gain:

```csharp
bool IncludeGenerated   = false;            // filter *.g.cs, obj/, bin/
string? MinimumSeverity = "Warning";        // Error | Warning | Info | Hidden
string[]? ExcludeDiagnosticCodes;           // ["IDE0005", "CS8019"]
string[]? ExcludeDiagnosticSources;
```

**New tools** (`find_entrypoints`, `find_registrations`, `find_dead_code_candidates`, `test_map`) gain where semantically meaningful:

```csharp
bool IncludeTests       = <tool-specific default>;
string[]? ExcludePaths;                     // path-fragment substring match
int MaxResults          = <tool-specific default>;
```

Filter implementation is pure post-filter at serialization time — never affects index construction. This keeps caching layer simple.

### Index-population trigger

The same parallel-per-project warm-up loop that builds `SymbolIndex` is extended to also build `InvocationIndex`. One pass over each project's syntax trees + semantic model. `WarmupTask` completes only when both indexes finish (consistent with v1.2).

## Tool specs

### `find_entrypoints`

**Input:**
```csharp
record FindEntrypointsInput(
    bool IncludeAspNetRoutes      = true,
    bool IncludeHostedServices    = true,
    bool IncludeMiddlewarePipeline= true,
    int MaxResultsPerSection      = 20,
    string Format                 = "structured");
```

**Output (structured):**
```json
{
  "routes": [
    { "verb": "GET", "template": "/api/users/{id}", "handler": "GetUserAsync",
      "location": { "filePath": "...", "line": 42, "column": 9 } }
  ],
  "middleware": [
    { "method": "UseAuthentication", "location": {...} }
  ],
  "hostedServices": [
    { "kind": "registered", "service": "MyApp.Workers.EmailWorker", "location": {...} },
    { "kind": "subclass", "type": "MyApp.Workers.PollingWorker",
      "baseType": "BackgroundService", "location": {...} }
  ],
  "truncated": []
}
```

**Backing data:** `InvocationIndex._routes` / `_middleware` / `_hostedServices`. Sub-100 ms once warm-up is done.

**Detection patterns** (syntactic):
- Routes: `InvocationExpressionSyntax` where method name ∈ `{MapGet, MapPost, MapPut, MapDelete, MapPatch, MapMethods, Map}` AND receiver is `IEndpointRouteBuilder` (via semantic model). Verb extracted from method name; template from first string-literal argument.
- Middleware: `InvocationExpressionSyntax` where method name starts with `Use` AND receiver is `IApplicationBuilder` or `WebApplication`.
- Hosted services: `AddHostedService<T>()` calls + `INamedTypeSymbol` instances whose `BaseType.Name == "BackgroundService"` walked from `SymbolIndex` data.

### `find_registrations`

**Input:**
```csharp
record FindRegistrationsInput(
    string? Query              = null,    // service-type / impl-type / raw text fragment
    bool IncludeConsumers      = true,
    int MaxResults             = 20,
    string Format              = "structured");
```

**Output (structured):**
```json
{
  "registrations": [
    { "serviceType": "IFooService", "implType": "FooService",
      "lifetime": "Singleton",
      "location": { "filePath": "...", "line": 12, "column": 9 },
      "rawCall": "services.AddSingleton<IFooService, FooService>()",
      "likelyConsumers": [
        { "type": "BarController", "ctorLocation": {...} }
      ]
    }
  ],
  "unclassified": [
    { "rawCall": "services.AddMyCustomBootstrap()", "location": {...} }
  ],
  "truncated": []
}
```

**Detection patterns**:
- `Add{Singleton,Transient,Scoped}<TService, TImpl>()` — both type args available via semantic model.
- `Add{Singleton,Transient,Scoped}<TService>(factory)` — only service type.
- `Add{Singleton,Transient,Scoped}(typeof(...), typeof(...))`.
- `services.AddX()` extension calls where receiver is `IServiceCollection` and method name is NOT one of the recognised forms → `unclassified[]`. Agent stays informed about uncategorized DI surface area.

**Consumer detection** (when `IncludeConsumers: true`): for each registration's `serviceType`, walk symbols whose constructor parameter type matches. Reuses `SymbolIndex._byParameterType` — already indexed in v1.2.

### `project_overview`

**Input:**
```csharp
record ProjectOverviewInput(
    int MaxProjects                  = 25,
    int MaxPackagesPerProject        = 8,
    int MaxProjectReferencesPerProject = 8,
    string Format                    = "structured");
```

**Output (structured):**
```json
{
  "solutionPath": "C:\\projects\\duetgpt\\duetGPT.sln",
  "projects": [
    { "name": "duetGPT", "targetFramework": "net10.0",
      "documentCount": 142,
      "projectReferences": ["duetGPT.Module"],
      "packageReferences": [ { "name": "DevExpress.ExpressApp", "version": "24.2.3" } ]
    }
  ],
  "diagnostics": [...]  // forwarded from WorkspaceService.Diagnostics (v1.1)
}
```

**Implementation:** direct read off `Solution.Projects`, `Project.MetadataReferences` filtered to NuGet, `Project.ProjectReferences`. No new index. Sub-50 ms.

### `analyze_symbol`

**Input:**
```csharp
record AnalyzeSymbolInput(
    string? SymbolId,
    string? FilePath,
    int? Line,
    int? Column,
    AnalyzeSymbolIncludes? Include,
    int MaxPerSection = 20,
    string Format     = "structured");

record AnalyzeSymbolIncludes(
    bool Hover           = true,
    bool References      = true,
    bool Implementations = true,
    bool DerivedTypes    = true,
    bool Callers         = true);
```

**Output (structured):**
```json
{
  "symbol": { "id": "M:Foo.Bar.Baz", "kind": "Method", "displayName": "...",
              "containingType": "Foo.Bar" },
  "hover":           { "signature": "...", "documentation": "..." } | null,
  "references":      { "count": 7, "items": [...] } | null,
  "implementations": { "count": 2, "items": [...] } | null,
  "derivedTypes":    { "count": 0, "items": [] } | null,
  "callers":         { "count": 4, "items": [...] } | null,
  "truncated":       ["references"]
}
```

**Implementation:** resolve symbol once via `RoslynHelpers.ResolveSymbol`. In parallel call the lower-level helpers that the existing tools wrap — **not** the tool classes (no nested `ToolResult<T>`). Sections that don't apply to the symbol kind (e.g., `derivedTypes` on a method) return `null`, not absent. Sections explicitly disabled via `Include.X = false` also return `null`. `truncated[]` lists section names that hit `MaxPerSection`.

### `find_callees`

**Input:** identical to `find_callers`.

**Output:** identical shape — `items[]` of `{ symbol, location }` representing methods invoked from the target method's body.

**Implementation:**
1. Resolve target symbol; must be `IMethodSymbol` (else `SYMBOL_NOT_FOUND` with hint).
2. Walk `MethodDeclarationSyntax`'s body — collect `InvocationExpressionSyntax` and `ObjectCreationExpressionSyntax`.
3. For each, ask `SemanticModel.GetSymbolInfo` → `IMethodSymbol`.
4. Dedupe by `DocumentationCommentId`, sort by source-order, cap at `MaxResults`.

No index needed — operates on a single method body. Typical cost < 50 ms.

### `test_map`

**Input:**
```csharp
record TestMapInput(
    string? SymbolId,
    string? FilePath,
    int? Line,
    int? Column,
    int MaxResults = 10,
    string Format  = "structured");
```

**Output (structured):**
```json
{
  "candidates": [
    { "testSymbol": "Foo.Tests.FooTests.Compute_returns_42",
      "location": {...},
      "confidence": "high",
      "via": "reference" }
  ],
  "testProjectsScanned": ["Foo.Tests", "Foo.IntegrationTests"]
}
```

**Algorithm:**
1. Resolve production symbol. Find its containing project `P`.
2. Walk `Solution.Projects` — keep projects that (a) transitively reference `P` AND (b) match test heuristic: name ends in `.Tests`/`.Test`/`.IntegrationTests` OR has a `MetadataReference` whose name matches `xunit*` / `nunit*` / `MSTest.TestFramework`.
3. Run `SymbolFinder.FindReferencesAsync` on the symbol, keep only locations whose document is in a test project from step 2 → **high-confidence** with `via: "reference"`.
4. For each test project, walk symbols whose name matches `{Name}Tests` / `{Name}Spec` / `Test{Name}` / `Tests{Name}` (where `{Name}` is the simple name of the production symbol or its containing type) → **medium-confidence** with `via: "name-match"` if not already seen.
5. Sort high → medium, dedupe by `(filePath, line)`, cap at `MaxResults`.

Returns `{ candidates: [] }` on no matches — empty is not failure.

### `find_dead_code_candidates`

**Input:**
```csharp
record FindDeadCodeInput(
    bool IncludePrivateMembers = true,
    bool IncludeInternalTypes  = true,
    bool IncludeTests          = false,
    int MaxResults             = 20,
    string[]? ExcludePaths     = null,
    string Format              = "structured");
```

**Output (structured):**
```json
{
  "candidates": [
    { "symbol": "Foo.Bar.UnusedMethod", "kind": "Method",
      "accessibility": "Private",
      "location": {...},
      "confidence": "high",
      "reason": "no-references" }
  ],
  "projectsScanned": ["Foo.Core", "Foo.Web"],
  "skipped": { "publicMembers": 142, "tests": 87, "denylisted": 12 }
}
```

**Algorithm:**
1. Walk `SymbolIndex` entries with accessibility in `{ Private }` (when `IncludePrivateMembers`) or `{ Internal }` for types and methods on internal types (when `IncludeInternalTypes`).
2. Skip symbols matching the **denylist**:
   - Attributed with any of: `[Fact]`, `[Theory]`, `[Test]`, `[TestMethod]`, `[Benchmark]`, `[JsonConstructor]`, `[OnDeserialized]`, `[OnDeserializing]`, `[ModuleInitializer]`, `[UnmanagedCallersOnly]`, `[DllImport]`.
   - Type implements `IXmlSerializable`, `ISerializable`.
   - Member name is `Dispose`, `DisposeAsync`, `ToString`, `Equals`, `GetHashCode` (implicit framework contracts).
   - Constructor with zero params on a type that has `[Serializable]` or is a record.
3. For internal types/methods: check `Compilation.Assembly.GetAttributes()` for `[InternalsVisibleTo("…")]`. If present, internal members of that assembly are effectively friend-visible — assign `confidence: "medium"` not `"high"`. (Public access modifier check happens before this; we don't index public members.)
4. For each surviving candidate, run `SymbolFinder.FindReferencesAsync`. If `Locations.IsEmpty`:
   - Private member → `confidence: "high"`, `reason: "no-references"`.
   - Internal member, no `[InternalsVisibleTo]` → `confidence: "high"`.
   - Internal member with `[InternalsVisibleTo]` → `confidence: "medium"`, `reason: "no-references-but-internals-visible-to-friends"`.

Expected cost on duetGPT: 2–5 s (linear in private/internal symbol count × cost-per-`FindReferencesAsync`). Acceptable — not interactive-loop tool. Result count gives skipped totals so the agent can sanity-check coverage.

## Public surface change

`IWorkspaceService` gains one property (mirrors v1.2's `SymbolIndex`):

```csharp
InvocationIndex InvocationIndex { get; }
```

Both `SymbolIndex` and `InvocationIndex` types remain `public sealed` (interface forces this; same trade-off accepted in v1.2). Consumers are expected to treat them as implementation details — direct dependency only inside `WorkspaceService` and the tool classes.

A shared `HashSet<DocumentId> _dirty` is added on `WorkspaceService`. Both indexes' `MarkDirty(DocumentId)` is called from a single point in `GetFreshSolutionAsync`. (v1.2's per-index dirty set is consolidated.)

## Performance expectations

| Phase | v1.2 | v1.3 |
|---|---|---|
| `LoadAsync` return | ~2 s (unchanged) | unchanged |
| Warm-up total (background, duetGPT) | ~12–16 s | ~12.5–16.5 s (+200–400 ms) |
| `find_entrypoints` cold-cache | n/a | <100 ms |
| `find_registrations` cold-cache | n/a | <100 ms |
| `project_overview` | n/a | <50 ms (no index) |
| `analyze_symbol` typical | n/a | <500 ms (5 parallel sub-queries, dominated by `FindReferencesAsync`) |
| `find_callees` | n/a | <50 ms |
| `test_map` | n/a | <300 ms (one `FindReferencesAsync` + symbol-name scan) |
| `find_dead_code_candidates` | n/a | 2–5 s (linear in private/internal count) |
| Memory cost of `InvocationIndex` | n/a | ~few MB on duetGPT-sized solutions; small payload per entry |

Index-build cost is dominated by the existing symbol walk; adding syntax-tree invocation collection in the same pass adds proportionally little.

## Risks

- **Detection patterns are syntactic.** Custom extension methods on `IServiceCollection` won't be classified into lifetime buckets unless they match the known forms. Mitigation: `unclassified[]` array surfaces them so agents aren't blind. Living list.
- **Test-project heuristic false positives.** A non-test project named `Acme.IntegrationTests.Helpers` would match. Mitigation: combined with framework-package check (xunit/nunit/MSTest) for higher precision; document the heuristic in the tool description.
- **Dead-code reflection blindness.** Roslyn cannot see `Type.GetMethod("…").Invoke(...)` style calls, or framework conventions like ASP.NET Razor `OnGet` discovery. Mitigation: denylist + accept medium-confidence false-positive rate; document explicitly. Agent should always verify before deleting.
- **`InvocationIndex` parallel-build contention.** Single shared lists with `lock`-based appends. If contention shows up on >16-project solutions, switch to per-project lists merged after. YAGNI for now.
- **`format: "summary"` lossy for some tools.** Tools where the structured output carries information the summary can't (e.g., `find_references` summary "7 references in 4 files" loses which files). Acceptable — `summary` is opt-in.
- **Breaking shape on existing tools.** `format` and filter params are additive with defaults that produce identical output to v1.2. No existing input is invalidated. Verified by `FormatParameterTests` byte-identical-structured assertion.

## Open questions / deferred

- **Surface invocation-index health via `reload_workspace` output.** Like the SymbolIndex `buildDurationMs` nice-to-have from v1.2 — add as a follow-up if useful.
- **`semantic_search` new patterns.** csharp-lsp-mcp also has `config-bindings:` (`IConfiguration.GetSection(...).Bind(...)`) and `middleware-pipeline:` semantic search modes. We're folding middleware into `find_entrypoints` and could add a `config-bindings:` pattern to `semantic_search` later if duetGPT/XAF sessions show demand. Defer.
- **`change_impact` revisit.** If Claude Code stops doing planning in-loop well at some scale, revisit a server-side composite. Not now.
- **Persistent on-disk index.** A `.mcpRoslyn-cache/` directory could persist `SymbolIndex` + `InvocationIndex` across process restarts. Useful only if cold-start cost (warm-up ~12–16 s on duetGPT) becomes a session-startup pain point. Defer until observed.
- **Cross-platform** still deferred from v1 design.

## Sequencing

Single release v1.3. Implementation plan (separate doc) sequences the changes to keep the working tree green:

1. Shared `_dirty` consolidation on `WorkspaceService` (refactor; no behavior change).
2. `InvocationIndex` skeleton + tests passing on fixture.
3. New fixture projects (`TestSolution.Web`, `TestSolution.Tests`) + dead-code injections.
4. Tool: `project_overview` (no index dependency; landed first as a confidence-builder).
5. Tool: `find_entrypoints`.
6. Tool: `find_registrations`.
7. Tool: `find_callees`.
8. Tool: `analyze_symbol`.
9. Tool: `test_map`.
10. Tool: `find_dead_code_candidates`.
11. Cross-cutting: `format` parameter (all tools).
12. Cross-cutting: filter knobs on diagnostics tools.
13. Doc updates (`ARCHITECTURE.md`, `TODO.md`, `SESSION_HANDOFF.md`).
14. Manual acceptance run on duetGPT → `docs/acceptance/2026-05-20-v1.3-acceptance.md`.

Each step is independently mergeable. Tests stay green throughout.
