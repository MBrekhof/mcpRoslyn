## Warm-up cost delta

- v1.2 baseline: ~22.8 s on duetGPT (compile + SymbolIndex build)
- v1.3 predicted: +200–400 ms (InvocationIndex on the same trees)
- v1.3 actual: **~31 s total warm-up** = 7.5 s compile + 10.5 s SymbolIndex + **13.0 s InvocationIndex**
- ❌ **Prediction was off by ~30x.** InvocationIndex added ~13 s, not the predicted 200–400 ms. This is the most significant performance finding of v1.3.

Cross-check against earlier server-startup runs in the same log (4:50 and 5:04 cold starts of the server): InvocationIndex took 7.2 s in both — still 18–36x over the 200–400 ms prediction even on faster runs. The 13 s in the active acceptance run is the upper end of variance, but the order of magnitude is consistent.

## Issues surfaced

1. **InvocationIndex warm-up cost is ~13 s, predicted +200-400 ms.** ~30x over budget. Investigate whether InvocationIndex can use the parsed syntax trees from SymbolIndex rather than re-parsing, or whether incremental population is feasible.

2. **`find_references` cold-cache call regressed 2.8x** (641 ms v1.2 → 1803 ms v1.3) on the same `IKnowledgeService` symbol. Warm timing is fine. Suggests v1.3 added per-symbol work on the first lookup — possibly InvocationIndex consultation that should be lazy.

3. **`find_implementations` regressed 8.4x** (321 ms v1.2 → 2682 ms v1.3) on the same interface. This is a single call (not a cold/warm comparison) — the v1.3 implementation appears materially slower regardless of cache state. Highest-priority regression to investigate.

4. **`find_references` returns inconsistent counts across calls** for the same symbol — first call returned 35 results, immediate second call returned 34. Same input, no workspace changes between calls. Likely a dedup/ordering bug in the new InvocationIndex-integrated reference scan. Worth a unit test.

5. **`find_dead_code_candidates` false-positive rate is high on Blazor `.razor.cs` files.** Event handlers (`EditUser_Click`, `SaveQuotaChanges`, etc.) and `[Inject]` properties are wired up from Razor markup, which mcpRoslyn doesn't see. Recommend: detect classes deriving from `Microsoft.AspNetCore.Components.ComponentBase` and either skip or down-weight to medium confidence.

6. **Default truncation hits silently on hot tools** (`find_entrypoints` middleware section, `find_registrations` registrations + unclassified, `analyze_symbol` references at 20/34). The `truncated:[...]` flag is present in the response but easy to miss. Consider raising default `maxResults`/`maxResultsPerSection`, or surfacing truncation more prominently in the structured output.

7. **`semantic_search has-attribute:McpServerToolType` returned 0 matches against duetGPT** — that attribute is mcpRoslyn-internal. This is correct behavior, but the acceptance doc baseline came from running against mcpRoslyn itself, not duetGPT. Worth swapping the baseline query for one that exercises duetGPT (e.g., `has-attribute:Microsoft.AspNetCore.Mvc.HttpGetAttribute` or `implements:Microsoft.EntityFrameworkCore.IDbContextFactory`).

8. **Baseline `duetGPT` project carries 622 compile-time diagnostics** including hard errors (`CS0234` Pages namespace missing, `CS0246` App type missing, `CS0103` StateHasChanged missing, `CS0115` no method to override, `CS0117` `DbContext.Documents` missing). These are not v1.3 regressions and were present in v1.2 too. Root cause: mcpRoslyn doesn't trigger the Blazor source generators (the `.razor` markup emits partial classes that contain `App`, `StateHasChanged` via `ComponentBase`, the `Pages` namespace, and `DbContext.Documents` declaration). This degrades accuracy of dead-code detection (issue #5) and any tool that walks the type graph. Likely a deeper MSBuild integration fix in a future release; not blocking v1.3.

## Verdict

**PASS-WITH-FOLLOWUPS**

All seven new v1.3 tools are functional and correct on real-world inputs. Filter knobs and the new `format:"summary"` path both work. Project counts, endpoint accuracy, registration consumer tracing, callee enumeration, and test mapping are all spot-checked accurate.

However, three real performance regressions vs the v1.2 baselines (issues #1, #2, #3) need follow-up before this can be called a clean release:
- InvocationIndex warm-up ~30x over predicted budget
- `find_references` cold path 2.8x slower
- `find_implementations` 8.4x slower

Plus a correctness signal worth chasing (issue #4 — inconsistent `find_references` counts) and an ergonomic improvement for Blazor codebases (issue #5).

None of these are catastrophic — the tool is usable today — but they should be addressed in a v1.3.1 patch before promoting v1.3 as the recommended release.
