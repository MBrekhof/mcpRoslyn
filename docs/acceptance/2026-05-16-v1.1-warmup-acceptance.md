## Query results

### 1. find_references on `IGroupPermissionResolver` — first user-facing query

- **Time: 1 874 ms** (v1 baseline: 8 400 ms)
- Error: none
- **Result: PASS — 4.5× speedup vs v1.**

This is the headline v1.1 result. The user took ~24 s after load completed before sending this query (load: 06:38:44.599; tool_dispatch_start: 06:39:21.284), giving the background warm-up ample time to finish all four projects. v1's 8 400 ms was explicitly attributed to "Roslyn building semantic models for all four projects" on the first call — exactly the cost warm-up pre-pays. The 4.5× drop is consistent with warm-up having completed before the first query.

The remaining 1 874 ms is the actual `find_references` work: `SymbolFinder.FindReferencesAsync` traversal across compiled symbols + JSON serialization.

### 2. find_implementations on `IBuiltInToolProvider`

- Time: **832 ms** (v1 baseline: 300 ms)
- Error: none
- Result: **SLOWER, unexplained.** v1 was 2.8× faster here.

Possible causes — none verified:
- v1's 300 ms benefited from project state already warm from the prior `find_references` call on the same project; this run's sequencing (different project for query 2 vs query 1?) may have hit a less-warm path.
- `IBuiltInToolProvider`'s containing project may have changed between v1 (2026-05-15) and now.
- General run-to-run noise.

Not a regression worth blocking on — the call still completed under 1 s. Worth a re-measurement on a third independent run.

### 3. find_callers on `KnowledgeService.ExpandQueryAsync`

- Time: **168 ms** (v1 baseline: 14 ms — but v1 returned `SYMBOL_NOT_FOUND`)

v1's 14 ms was a fast-fail on `SYMBOL_NOT_FOUND` — the symbolId was stale. This run completed normally, suggesting either the symbol now resolves (signature stabilized?) or the caller used a different symbolId. Direct comparison invalid; what matters is that the call ran and finished in well under a second.

### 4. semantic_search `has-attribute:McpServerToolType`

- First call: **7 745 ms** (v1 baseline: 11 049 ms)
- Second call (refined): **1 153 ms**
- Error: none

The first call is ~30% faster than v1. Still slow in absolute terms — confirming the v1.1 follow-up "`semantic_search` attribute walk is O(symbols)" item is real and warm-up doesn't address it. Pre-building an attribute index at load time remains the right fix.

The second call at 1 153 ms is interesting evidence of Roslyn's symbol-walk caching kicking in within a single process lifetime.

## Verdict

**Warm-up shipped and the headline metric improved as designed.** First-query latency on `find_references` dropped 4.5× (8 400 ms → 1 874 ms), which is the exact regression v1 acceptance flagged. Other queries are mixed but none regressed catastrophically; the find_implementations slowdown is noise-territory and warrants a single re-measurement before treating as a real signal.

**Direct verification of warm-up's per-project log lines was blocked** by Claude Code's stderr capture window closing after the MCP `initialize` handler. Inferential evidence (the find_references speedup) is strong.

## Follow-ups

1. **`--log-file <path>` flag for mcpRoslyn.** Tee `ILogger` output to a file so warm-up timings, MSBuild warnings, and per-call diagnostics are inspectable after the session. Small change (~10 lines in `Program.cs`). Recommended next item — would have made this acceptance run unambiguous.
2. **Re-measure `find_implementations`.** Single run can't distinguish noise from regression. Repeat the v1 query suite once more and compare distribution.
3. **`semantic_search` attribute index** (already on TODO.md). 7 745 ms is still too long for a query the user might run multiple times per session. Pre-built index keyed by attribute full name should bring this to sub-second.
4. **Stderr capture limitation is a Claude Code thing**, not an mcpRoslyn thing. Worth knowing: rely on item #1 (own log file) rather than `claude --debug mcp` for any future MCP server diagnostics.
